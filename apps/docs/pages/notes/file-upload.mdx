# 大檔案上傳：前端視角的實作筆記

> 參考：[File upload (notes) — Tiger Abrodi](https://tigerabrodi.blog/file-upload-notes)

---

## 為什麼不能直接 POST 到後端？

傳統做法是把檔案直接送到自己的 server，但遇到大檔就會炸：

- **Timeout**：HTTP 連線超時，尤其是行動裝置或網路不穩的情境
- **Server 壓力**：每個上傳請求都佔用 server 資源與頻寬
- **斷線就要重來**：沒有任何斷點續傳能力

現代做法是讓 client **直接上傳到 S3**，server 只負責「核發通行證（pre-signed URL）」和「記錄結果」，完全不碰檔案本身。

---

## 前端的完整流程

### Phase 1：跟 server 拿 upload 許可

```ts
// 1. 通知 server 準備接收這個檔案，拿到 file_id
const { fileId } = await api.initiateUpload({
  name: file.name,
  size: file.size,
  totalChunks: Math.ceil(file.size / CHUNK_SIZE),
})
```

Server 這時會在 DB 建立 `files` 和 `file_chunks` 的 pending 記錄，並回傳 `file_id`。

---

### Phase 2：把檔案切成 chunks

```ts
const CHUNK_SIZE = 5 * 1024 * 1024 // 5MB（S3 multipart 的最小 chunk 要求）

function splitFileIntoChunks(file: File): Blob[] {
  const chunks: Blob[] = []
  let start = 0

  while (start < file.size) {
    chunks.push(file.slice(start, start + CHUNK_SIZE))
    start += CHUNK_SIZE
  }

  return chunks
}
```

**為什麼是 5MB？** 這是 S3 multipart upload 對 chunk 的最小限制（最後一個 chunk 除外）。

---

### Phase 3：取得每個 chunk 的 pre-signed URL

```ts
// 可以一次拿全部，也可以分批拿（分批比較不容易 timeout）
const { presignedUrls } = await api.getPresignedUrls({
  fileId,
  chunkNumbers: chunks.map((_, i) => i + 1), // S3 part number 從 1 開始
})
```

Server 回傳的是帶有時效的 URL，格式類似：

```
https://s3.amazonaws.com/bucket/uploads/{fileId}/chunks/{chunkNumber}?X-Amz-Expires=...
```

---

### Phase 4：逐一上傳 chunk，收集 ETag

```ts
interface ChunkResult {
  partNumber: number
  etag: string
}

async function uploadChunk(
  url: string,
  chunk: Blob,
  partNumber: number,
  onProgress: (loaded: number) => void,
): Promise<ChunkResult> {
  return new Promise((resolve, reject) => {
    const xhr = new XMLHttpRequest()

    xhr.upload.onprogress = (e) => onProgress(e.loaded)

    xhr.onload = () => {
      if (xhr.status === 200) {
        // S3 在 response header 裡回傳 ETag，最後 complete 時需要用到
        const etag = xhr.getResponseHeader('ETag') ?? ''
        resolve({ partNumber, etag })
      } else {
        reject(new Error(`Chunk ${partNumber} failed: ${xhr.status}`))
      }
    }

    xhr.onerror = () => reject(new Error(`Network error on chunk ${partNumber}`))

    xhr.open('PUT', url)
    xhr.send(chunk)
  })
}
```

**為什麼用 XHR 而不是 fetch？** `fetch` 目前沒有穩定的 upload progress API，XHR 的 `xhr.upload.onprogress` 是處理上傳進度最直接的方式。

---

### Phase 5：通知 server 完成

```ts
await api.completeUpload({
  fileId,
  parts: chunkResults, // [{ partNumber: 1, etag: '...' }, ...]
})
```

Server 收到後會：

1. 比對 DB 裡所有 chunk 是否都已完成
2. 呼叫 S3 的 `CompleteMultipartUpload` API（需要所有 part 的 ETag + partNumber）
3. 更新 `files` 狀態為 `completed`

## 前端需要管理的狀態

大檔上傳的 UI 複雜度主要在**狀態追蹤**。

```ts
interface ChunkState {
  partNumber: number
  status: 'pending' | 'uploading' | 'done' | 'error'
  etag?: string
  uploadedBytes: number
}

interface UploadState {
  fileId: string
  totalSize: number
  chunks: ChunkState[]
  status: 'idle' | 'uploading' | 'completing' | 'done' | 'error'
}
```

整體進度的計算：

```ts
function calcProgress(state: UploadState): number {
  const uploaded = state.chunks.reduce((sum, c) => sum + c.uploadedBytes, 0)
  return Math.round((uploaded / state.totalSize) * 100)
}
```

## 並發控制：不要同時送太多

一次開幾十個並發上傳會打爛瀏覽器的連線限制，也容易造成網路擁塞。建議控制在 3～5 個並發。

```ts
async function uploadWithConcurrency<T>(tasks: (() => Promise<T>)[], concurrency: number): Promise<T[]> {
  const results: T[] = []
  const executing = new Set<Promise<void>>()

  for (const task of tasks) {
    const p = task().then((result) => {
      results.push(result)
      executing.delete(p)
    }) as Promise<void>

    executing.add(p)

    if (executing.size >= concurrency) {
      await Promise.race(executing)
    }
  }

  await Promise.all(executing)
  return results
}
```

## 錯誤處理與重試

單個 chunk 失敗不應該讓整個上傳失敗，應該針對失敗的 chunk 重試：

```ts
async function uploadChunkWithRetry(
  url: string,
  chunk: Blob,
  partNumber: number,
  maxRetries = 3,
): Promise<ChunkResult> {
  let lastError: Error

  for (let attempt = 1; attempt <= maxRetries; attempt++) {
    try {
      return await uploadChunk(url, chunk, partNumber, () => {})
    } catch (e) {
      lastError = e as Error
      // exponential backoff
      await new Promise((r) => setTimeout(r, 1000 * attempt))
    }
  }

  throw lastError!
}
```

## 斷點續傳

如果把 `UploadState` 持久化到 `localStorage` 或 `IndexedDB`，就可以在使用者重新整理或網路中斷後從上次失敗的 chunk 繼續：

```ts
// 儲存進度
localStorage.setItem(`upload:${fileId}`, JSON.stringify(uploadState))

// 恢復進度：跳過已完成的 chunk
const savedState = JSON.parse(localStorage.getItem(`upload:${fileId}`) ?? 'null')
const pendingChunks = savedState?.chunks.filter((c) => c.status !== 'done') ?? allChunks
```

Pre-signed URL 有時效限制，如果是隔天繼續，需要重新跟 server 拿新的 URL，但 ETag 和 chunk 完成狀態可以從 DB 恢復，不需要重傳已完成的 chunk。

## 上傳驗證：誰來確認 chunk 到了？

S3 chunk 上傳成功後，有兩種方式確認：

| 方式                            | 說明                             | 可靠性                              |
| ------------------------------- | -------------------------------- | ----------------------------------- |
| **Client-driven**               | Client 上傳成功後主動通知 server | ❌ Client 可能在通知前 crash 或偽造 |
| **Storage-driven（S3 Events）** | S3 自動透過 SNS/SQS 通知 server  | ✅ 更可靠，無法偽造                 |

對前端來說，這個差別主要影響 UI 的設計：如果是 S3 Events 驅動，完成請求後可能需要 polling 或 WebSocket 等待 server 確認所有 chunk，而不是直接信任 client 的回報。

## 實作 checklist

- [ ] `file.slice()` 切分 chunk（不會把整個檔案 load 進記憶體）
- [ ] XHR 處理上傳進度
- [ ] 並發控制（3～5 個）
- [ ] 收集每個 chunk 的 ETag
- [ ] 單 chunk 失敗重試 + exponential backoff
- [ ] 整體進度計算與 UI 呈現
- [ ] 上傳狀態持久化（斷點續傳）
- [ ] Pre-signed URL 過期處理（重新取得）
- [ ] 使用者取消：呼叫 `xhr.abort()`，通知 server 清理未完成的 multipart upload
