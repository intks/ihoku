## 什麼是 Code Review？

Code Review 的目標不是追求完美的程式碼，而是確保**每一次變更都明顯提升、或至少不降低整體程式碼健康度**。

它同時也是一個雙向的協作過程：Reviewer（審查者）負責守住品質與一致性，Author（提交者）則透過良好的準備，讓整個流程順暢、有效率。兩者缺一不可。

---

## Part 1｜Reviewer 的職責

### 看什麼？

Reviewer 在檢視一個 CL（changelist / PR）時，主要關注：

- **設計**：是否合理？與整體架構、模組邊界是否契合？
- **功能**：是否符合預期？有無並發問題（race condition、deadlock）？
- **複雜度**：有沒有過度設計？能不能拆小、簡化？
- **測試**：有沒有搭配正確且有意義的自動化測試（而不是為寫而寫）？
- **命名**：是否清楚、精煉？
- **註解**：說的是「為什麼這樣做（why）」，而不只是「在做什麼（what）」？
- **風格**：是否符合團隊規範？純粹個人偏好只作非強制建議（標上 `Nit:`）。
- **文件**：有行為變更時，是否同步更新 README / 文件？

### 怎麼看大型 CL？

1. 先讀 CL 描述，判斷這個改動是否有意義。
2. **優先看關鍵設計**，若設計本身有大問題，及早給出回饋，其他細節暫不看。
3. 確認主體沒問題後，再按邏輯順序看完剩餘變更。

遇到真的看不懂或風險高（安全、並發、國際化）的部分，必要時找更資深的 Reviewer，或請作者當面解釋。

### 速度

- **一個工作日內**給出回應（不是要求一天通過，而是快速回覆評論）。
- 跨時區時，盡量在對方上班時段內給出意見。
- 若剩餘只是小問題，可用「LGTM with comments」：先批准，並說明預期的後續小調整，避免卡住進度。

### 大型 CL 的處理

可以直接要求作者拆成多個較小 CL。若實在無法拆分，至少先對整體設計給出意見，避免作者在錯誤設計上繼續疊加工作。

### 評論語氣

針對**程式碼**，不要針對人。

- ✅「這段邏輯如果在 X 情況下會有問題，因為…」
- ❌「你怎麼會這樣寫…」
- 解釋「為什麼」這樣不好，幫助對方學習，而不只是「這樣不行」。
- 給出可能方向，但不需要替作者完成完整設計。
- 純教學、非強制建議，加上 `Nit:` 前綴，不阻塞 CL。

當作者 pushback 時，先檢查自己是否真的對，再以理由溝通；若分歧無法解決，升級給 TL / PM 決策。

## Part 2｜Author 的職責

### 黃金原則：尊重 Reviewer 的時間

Code Review 的大部分問題，都源自作者「沒把 CL 準備好就送出去」。Reviewer 不是個人的 QA 部門。

---

### 送出前：讓 PR 容易被 Review

#### 1. 先自己 Review 一遍

送出前，以 Reviewer 的角度把自己的 diff 從頭看一遍。常見收穫：

- 發現自己留了 debug log 或 TODO
- 找到自己忘記處理的邊界情況
- 確認命名和邏輯是否清楚

這一步能把最基本的問題過濾掉，讓 Reviewer 的時間花在真正有價值的地方。

#### 2. 寫清楚 CL 描述

一個好的 CL 描述包含兩件事：**做了什麼** 和 **為什麼這樣做**。

```
# ❌ 沒有資訊
Fix bug

# ✅ 清楚說明
限制 token 刷新的並發請求數

當多個請求同時偵測到 token 過期時，原本會同時發出多個刷新請求，
導致競態與不必要的 API 呼叫。改用 in-flight promise 共享機制，
確保同一時間只有一個刷新請求在執行。

相關 issue: #1234
```

- 第一行是簡潔的一句話摘要，後面留一行空行。
- Body 詳述：問題背景、解法理由、潛在缺點、相關 issue / 設計文件。
- 多次修改後送出前，確認描述仍然符合實際程式碼。

#### 3. 把機械式檢查自動化

如果你靠 Reviewer 指出縮排錯誤、沒過的測試、import 順序問題，是在浪費他們的時間。

- 設定 linter、formatter、pre-commit hook
- 確保 CI 全過才送 Review

Reviewer 應該把精力放在邏輯、設計、可讀性，而不是格式。

#### 4. 只做一件事，保持 CL 小而乾淨

**小 CL 的好處：**

- Review 更快、更完整，不容易漏看問題
- Bug 更少、更容易回滾
- Merge 衝突更少
- 審查者更願意抽出零碎時間來看

一般來說，約 200～400 行是舒服的規模，超過就考慮拆分。

**避免幾個常見的混亂：**

- 不要把格式改動（reformatting）混進功能改動裡。兩行功能修改被淹沒在幾百行的空白調整裡，是對 Reviewer 的一種不尊重。
- 不要把重構和行為改動放在同一個 PR。重構的 PR 應該有辦法用原本的測試直接驗證行為沒有改變。

**如何有序地拆分：**

```
Step 1：移動檔案、重命名（純重構 CL）
Step 2：改介面、改依賴（結構 CL）
Step 3：加入新功能（功能 CL，附對應測試）
```

依賴的多個 CL，要確保每一個被合入時都不會壞掉 build 或行為。

### Review 進行中：怎麼回應評論

#### 5. 用程式碼本身回答問題

當 Reviewer 對某段邏輯感到困惑，優先選擇**改程式碼讓它更清楚**，而不是在 Review 工具裡打一大段解釋。

```
Reviewer 問：這個函式在做什麼？

❌ 在 comment 裡回覆解釋
✅ 重新命名、拆分函式、或加入 code comment

原因：Review 討論只有你們看得到。
      未來閱讀程式碼的人，根本不會去翻 PR 歷史。
```

只有在程式碼和 code comment 都幫不上忙時，才用 Review 工具回覆解釋。

#### 6. 優雅地回應模糊評論

當評論很模糊，例如「這個函式很難懂」，不要只是「好我知道了」，也不要防禦性地反問「哪裡難懂？」

比較好的方式：

- 主動猜測 Reviewer 的意圖，先對程式碼做出修改。
- 如果不確定，用開放式提問：「你覺得怎樣會更好？」這比「你說的難懂是什麼意思」更不顯得防禦。

這種態度傳達的訊號是：**我願意改，告訴我方向**。

#### 7. Reviewer 不一定對——但要有耐心

如果你認為 Reviewer 理解有誤，先確認自己的解釋夠清楚，再禮貌地說明你的理由。

不要用憤怒回應評論。有情緒時先離開鍵盤，平靜後再回覆。

即便 Reviewer 語氣不好，也盡量只聚焦在技術問題上。若一再出現不尊重的評論方式，可以私下溝通；若沒有改善，再向主管反映。

#### 8. 主動釐清不清楚的地方

不確定 Reviewer 的意圖時，與其猜測、與其沉默，不如直接問。但問的方式很重要：

- ✅「你希望我在這裡怎麼調整？」
- ✅「我理解你的意思是 X，我試著把 Y 改成 Z，這樣有對嗎？」
- ❌「你說的不清楚，你到底是什麼意思？」

#### 9. 主觀的事，讓 Reviewer 贏

當某件事是「純粹的個人風格偏好」，沒有客觀對錯（例如：8 行的函式要不要拆成兩個 5 行的函式），通常應該**讓給 Reviewer**。

理由：Reviewer 對這段程式碼的閱讀體驗有直接感受。接受他們的建議，也表達了你對 Review 流程的尊重，建立長期的合作信任。

保留精力在有真正技術理由的地方堅持立場。

## 雙方共同的原則

| 原則                                       | Reviewer     | Author     |
| ------------------------------------------ | ------------ | ---------- |
| 目標是程式碼品質，不是「找碴」或「守面子」 | ✅           | ✅         |
| 只針對程式碼，不攻擊人                     | ✅           | ✅         |
| 快速回應，不讓對方等待                     | ✅           | ✅         |
| 說清楚「為什麼」，而不只說「這樣不行」     | ✅           | ✅         |
| 主觀偏好不擋 CL                            | ✅（標 Nit） | ✅（接受） |

---

## 結論

Code Review 的本質是**協作，不是對抗**。

Reviewer 的價值不在於找到最多問題，而在於幫助程式碼和作者一起成長。Author 的責任也不只是「把功能做完送審」，而是讓整個 Review 過程高效、清楚、有意義。

兩者都做好，Code Review 就不會是團隊的瓶頸，而是程式碼品質持續提升的引擎。
